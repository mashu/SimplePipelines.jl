Choosing operators, Workflow, and resources

Short map from “what you need” to the DSL piece to use.

Data flow between steps

GoalUse
Run A then B; B gets A’s single output (or last branch only if A had many)>> (sequence; see Composing pipelines)
Run A then B; B is a function step receiving all branch outputs as one vector|> (pipe)
Run A then B once per branch of A (each B sees one branch’s output).>> (broadcast >>)
Run A then B; B gets the same context as A (e.g. same ForEach wildcard), not A’s stdout>>> (same-input pipe)

When the left side has one successful path, >>, |>, .>>, and >>> behave the same for “what B receives” until you introduce Parallel / ForEach with multiple branches — then the table in Composing pipelines applies.

Rules vs hand-built DAG

SituationPrefer
Learn or debug one rule patterncheck on a @rule, then check(rule, "out/A.bam")
Many targets built from patterns (data/{id}.fq → out/{id}.bam), shared deps, Make-like resolve@workflow + @rule + @targets
One-off script, few steps, explicit graph in code>> / & on Steps directly
Inspect what would run without executingexplain for rule resolution, or print_dag on plan(wf)

Workflow is a registry of rules and default target strings; run(wf) calls plan then the same run as a Pipeline. See Rules and diagnostics for the gradual check → @workflow → explain path.

Memory and parallelism

SituationUse
Steps that reserve a lot of RAM or threads; cap concurrent usewith_resources on those nodes + run(..., memory_budget_mb=..., thread_budget=...)
Default laptop / workstation runDefaults (jobs, memory_budget_mb, auto_spill) are usually enough

See Running and inspecting for spill behaviour and the optional report callback on run.
